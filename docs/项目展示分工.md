## A：推荐算法（20-25分钟）

> 角色：技术负责人，展示推荐系统的架构设计、算法原理和效果

### 第一步：推荐系统全景架构（3分钟，口述 + 页面操作）

> 打开推荐页面 http://localhost:5173/#/recommend

口述要点（边说边在页面上指向对应区域）：
- 系统采用**三层架构**：召回层 → 融合层 → 重排序层
- **召回层**：5个并行模型各出 4n 个候选（n=最终推荐数量）
  - ItemCF（物品协同过滤）：听过A → 找和A相似的歌
  - UserCF（用户协同过滤）：找品味相似的用户 → 推荐他们喜欢的歌
  - ALS（矩阵分解）：播放行为 → 置信度矩阵 → 50维隐向量
  - Song2Vec（歌曲嵌入）：播放序列=句子 → Word2Vec → 100维向量
  - Sequence（序列推荐）：马尔可夫链 → N-gram回退
- **融合层**：Hybrid 加权融合，把5个模型的候选合并排序
- **重排序层**：偏好加权 + MMR多样性控制，输出最终结果
- 核心文件：engine.py（引擎）、reranker.py（重排序）、explainer.py（解释）、hybrid.py（融合）

**讲解要点**：不用单一模型是因为每个模型有各自的盲区——协同过滤在数据稀疏时效果差，序列模型在长尾用户上表现好但覆盖面窄，融合后取长补短。

### 第二步：混合推荐演示（3分钟）

1. 在"个性化推荐"Tab下，推荐模型下拉框选择"混合推荐"
2. 点击"获取推荐"按钮
3. 展示推荐结果列表，每首歌曲旁边有：匹配分数百分比、模型标签
4. 点击某首歌曲旁边的"i"按钮（推荐解释），弹出解释弹窗
5. 展示弹窗内容：匹配分数、来源模型列表、推荐理由文本
6. 点击"换一批"按钮，展示新一批推荐结果

**讲解要点**：每条推荐都附带可解释信息——告诉用户"为什么推荐这首歌"。来源模型标签说明这首歌曲是被哪些算法同时召回的。

### 第三步：单模型对比（4分钟）

1. 切换推荐模型为"物品协同过滤"，点击获取推荐，展示结果
2. 口述 ItemCF 核心思想：听过A→找和A相似的歌→推荐给你
   - 关键参数：k=50近邻、时间衰减（近期播放权重更高）、shrinkage修正（避免小样本偏差）
   - 代码位置：src/models/cf_model.py
3. 切换为"用户协同过滤"，获取推荐
4. 口述 UserCF 核心思想：找和你品味相似的用户→推荐他们喜欢的歌
   - 使用调整余弦相似度（减去用户均值，避免打分尺度差异）
5. 切换为"矩阵分解SVD"，获取推荐
6. 口述 ALS 核心思想：把播放次数转化为置信度矩阵，分解出用户和歌曲的50维隐向量
   - 参数：alpha=40（置信度放大系数）、lambda=0.1（正则化）、20次迭代
   - 优势：能发现隐式关联，不依赖直接的歌曲相似度
7. 切换为"歌曲嵌入"，获取推荐
8. 口述 Song2Vec 核心思想：把播放序列当句子、歌曲当词，用 Word2Vec 学习100维歌曲向量
   - 会话切分（60分钟间隔为一次会话）、时间加权用户画像
   - 优势：能捕捉"经常连续播放"的歌曲关系
9. 切换为"序列推荐"，获取推荐
10. 口述 Sequence 核心思想：基于马尔可夫链，预测"听完这首歌下一首可能听什么"
    - N-gram回退策略：3-gram→2-gram→1-gram→全局热门，Laplace平滑
    - 优势：命中率最高（~70-76%），尤其适合活跃用户

**讲解要点**：观察不同模型的推荐结果差异——ItemCF偏保守（推荐相似歌曲），Song2Vec偏发现（推荐关联歌曲），Sequence偏连贯（推荐下一步可能听的）。Hybrid 融合后取长补短。

### 第四步：Hybrid 融合机制深度讲解（3分钟，口述为主）

口述要点（可配合后台权重滑块演示）：

1. **Rank归一化**：5个模型输出的分数尺度不同（有的是相似度0-1，有的是播放次数），直接用分数相加不公平。所以先转成排名，再用 rank 归一化到 0-1 范围。

2. **加权融合**：默认权重 itemcf=35, usercf=20, svd=25, song2vec=10, sequence=10
   - ItemCF权重最高：因为"听过相似的歌"是最可靠的信号
   - Sequence权重最低：因为数据稀疏时N-gram命中率虽高但覆盖面窄

3. **一致性加成（Agreement Boost）**：如果一首歌被多个模型同时召回，额外加分。直觉是"大家都觉得好"的歌更值得推荐。

4. **10%探索注入**：最终结果中随机插入10%的非Top歌曲，避免推荐列表固化，让用户有机会发现新内容。

5. **热门歌曲惩罚**：CF和ALS模型中，对热门歌曲施加惩罚因子 1/(1+log(1+popularity))，避免推荐列表被大众热门歌曲占满。

> 在后台"模型"Tab展示"混合权重配置"区域，拖动某个模型的权重滑块，点击"保存权重"，说明管理员可以实时调整。

### 第五步：用户偏好画像构建（2分钟，口述 + 操作演示）

口述要点：
- 系统为每个用户构建偏好画像，数据来源包括：
  - 播放历史（listening_history 表）：哪些歌听过、听了几次、什么时候听的
  - 反馈数据（feedback 表）：喜欢/跳过/中性
  - 收藏数据（favorites 表）：收藏的歌曲
  - 偏好设置（user_preferences 表）：用户手动选择的偏好流派

- 画像维度：
  - 偏好歌手（喜欢的歌手加分+3）
  - 偏好流派（喜欢的流派加分+2）
  - 活跃时段（统计用户通常什么时候听歌）
  - 听歌多样性（听过多少不同歌曲/流派）

> 在用户端首页展示"你的偏好"标签区域和统计卡片，说明这些数据来源。

### 第六步：数据稀疏性挑战（2分钟，口述）

口述要点：
- 现实问题：12000+首歌、几十个用户，大部分歌曲只被少数人听过
- 数据稀疏度极高（>99%），传统协同过滤效果大打折扣
- 应对策略：
  - **Genre作为主信号**：在极稀疏数据下，流派是最稳定的推荐信号
  - **ALS替代SVD**：隐式反馈（播放次数→置信度）比显式评分更适合稀疏数据
  - **N-gram回退**：3-gram没数据就退到2-gram，再退到1-gram，最终退到全局热门
  - **Shrinkage修正**：协同过滤中，样本量小时相似度向0收缩，避免极端值
  - **Flat Boost重排序**：Top-5候选分数×20，是唯一在稀疏数据下有效的重排序策略

4. 选中3-5首歌曲
5. 点击"获取发现推荐"按钮
6. 展示基于选中种子生成的冷启动推荐结果

**讲解要点**：新用户没有听歌历史时，通过选择感兴趣的种子歌曲生成初始推荐，解决"新用户第一推什么"的问题。

### 第八步：推荐艺人与相似歌曲（2分钟）

1. 在推荐结果页面点击"推荐艺人"链接
2. 展示推荐艺人列表，每个艺人旁有"匹配 XX%"标签
3. 口述：基于用户听过的艺人频率和偏好画像推荐相似艺人
4. 进入任意一首歌曲的详情页
5. 向下滚动到"相似歌曲"区域，展示相似歌曲列表
6. 口述：相似歌曲基于歌曲向量的余弦相似度计算，和推荐系统是独立模块

### 第九步：在线重排序与过滤（2分钟，口述 + 操作）

口述要点：
- 模型输出的候选经过在线重排序（reranker.py）：
  - **偏好加权**：用户喜欢的歌手加分+3、喜欢的流派加分+2，不喜欢的风格减分
  - **MMR 多样性控制**：λ=0.7，70%相关性 + 30%多样性，避免推荐列表被同一种风格占满
  - **Flat Boost**：Top-5候选分数×20，突出最相关的推荐
- 自动过滤：已听过、已收藏、已喜欢、已跳过、黑名单中的歌曲不会出现在推荐中

> 获取一批推荐结果，观察列表中流派是否多样化，验证MMR的效果。

### 第十步：后台模型评估（3分钟）

> 切换到后台 http://localhost:8000/#/admin

1. 进入"模型"Tab
2. 展示训练范围选择：全部模型、混合推荐、物品协同过滤、用户协同过滤、矩阵分解SVD、歌曲向量、序列推荐
3. 展示数据切分方式：最近一次留出、随机留出、时间切分
4. 点击"仅评估"按钮，展示评估任务启动
5. 展示评估结果表格：各模型的 NDCG@100、命中率@100、覆盖率
6. 口述各模型的表现对比：
   - **Hybrid 综合最优**：NDCG ~57%, 命中率 ~66%（融合5个模型取长补短）
   - **Sequence 命中率最高**：~70-76%（马尔可夫链预测下一步很准）
   - **Song2Vec NDCG 最高**：~60-67%（向量空间中排序质量最好）
   - **ItemCF/UserCF**：数据稀疏时表现一般，但和ALS互补
7. 口述评估方法：leave-one-out（每个用户留出一首歌作为测试集，其余作为训练集）
8. 口述评估指标含义：
   - NDCG：推荐列表中好歌排在前面得分更高（排序质量）
   - 命中率：测试集中的歌是否出现在推荐列表里（召回能力）
   - 覆盖率：所有歌曲中被推荐过的比例（避免只推荐热门歌）

**讲解要点**：系统内置离线评估机制，评估结果证明混合推荐比单一模型效果更好。

**讲解要点**：批量歌词获取是异步后台任务，带进度追踪和取消机制，不影响前台使用。

### 第三步：语言识别策略（3分钟）

> 口述 + 操作演示

1. 口述4层语言识别策略（逐层回退，前一层能确定就不再往后）：
   - 第1层：流派关键词匹配（K-Pop→韩语，C-Pop→中文，J-Pop→日语，20+华语关键词...）
   - 第2层：歌词字符检测（CJK字符→中文/日文/韩文，Cyrillic→俄语，Devanagari→印地语...）
   - 第3层：拉丁字母回退（默认英语）
   - 第4层：国家代码推断（iTunes storefrontId → 国家 → 语言）

2. 口述识别优先级逻辑：
   - 有歌词时优先用歌词字符检测（最准确）
   - 无歌词时回退到标题和歌手名的字符检测
   - "Música Mexicana"特殊处理→西语而非葡语

> 在用户端 Browse 页面

3. 展示"语言"筛选栏：全部、中文、英文、日语、韩语、法语、西语、阿拉伯语、俄语、葡语、德语

6. 点击"中文"，展示筛选结果

7. 展示歌曲列表中每首歌的"语言分组"字段（ZH/EN/JA/KR...）

**讲解要点**：4层策略确保几乎所有歌曲都能被正确分类，歌词字符检测是最准确的一层。

> 在用户端

1. 进入 Browse 页面，找一首有封面的歌曲和一首无封面的歌曲对比
2. 展示无封面歌曲的渐变背景封面（根据流派生成不同颜色）
3. 进入"我的"→"歌单"Tab，展示歌单卡片的马赛克封面
4. 口述封面策略：
   - 有封面：直接使用 iTunes 封面图
   - 无封面：按流派生成渐变背景 + 音乐符号图标
   - 歌单封面：取歌单前4首歌的封面拼成2×2马赛克
   - 中文歌曲：不使用中文首字（避免乱码），统一显示音乐符号

### 第五步：数据质量保障（3分钟）

> 口述 + 代码/命令演示

1. **去重机制**（口述）：
   - 导入时按 iTunes trackId 去重（同一首歌在不同国家商店可能重复出现）
   - 按歌曲名+艺人名二次去重（不同 trackId 但实际是同一首歌的情况）
   - 数据库层面 tracks 表有 UNIQUE 约束兜底

2. **数据清洗**（展示终端命令）：
   - 运行 `python jobs/cleanup_unplayable_tracks.py`：清理没有试听地址的歌曲
   - 口述：导入的歌曲中可能有部分试听链接失效，清洗脚本自动移除这些无法播放的记录

3. **数据完整性检查**（展示终端命令）：
   - 运行 `python jobs/check_data_integrity.py`：全面检查数据质量
   - 口述检查项：
     - 外键引用完整性（评论引用的歌曲是否存在、播放记录引用的用户是否存在）
     - 必填字段非空检查（歌曲标题、艺人名不能为空）
     - 数据一致性（播放次数统计是否和 listening_history 表一致）

4. **数据库设计亮点**（口述）：
   - WAL 模式：支持并发读写，避免"database is locked"
   - busy_timeout=5000：读写锁等待5秒而非立即报错
   - 20张表，核心 tracks 表有20+字段，7个外键引用表
   - 删除一首歌需要级联清理7个关联表（收藏、评论、点赞、歌单、推荐日志、歌词缓存、画像缓存）

### 第六步：推荐缓存与离线计算（2分钟）

> 口述 + 终端演示

1. 口述推荐缓存机制：
   - 推荐模型训练和候选计算比较耗时（矩阵分解、向量计算）
   - `python jobs/build_recommend_cache.py` 预计算所有用户的推荐候选
   - 缓存存入数据库，用户请求推荐时直接读取缓存，响应更快
   - 数据更新后（新增播放记录、新导入歌曲）需要重新训练

2. 口述模型训练流程：
   - 后台"模型"Tab → 选择训练范围 → 点击"开始训练"
   - 训练过程有进度条，支持中途取消
   - 训练完成后自动更新评估指标
   - 支持增量训练（只训练指定模型）和全量训练

### 第七步：数据流全景总结（1.5分钟，口述）

口述要点（可以用手比划或简单板书）：

整个数据流是这样的：
1. **数据采集**：iTunes API → 40+国家商店 → 多风格关键词 → 去重导入
2. **数据增强**：歌词获取 → 语言识别 → 封面生成
3. **数据治理**：清洗脚本 → 完整性检查 → 外键约束
4. **模型训练**：播放记录 → 5个模型并行训练 → 缓存推荐候选
5. **在线服务**：用户请求 → 读取缓存 → 重排序 → 返回推荐结果

> "从数据采集到模型服务，整个链路保证了推荐结果的准确性和响应速度。"
